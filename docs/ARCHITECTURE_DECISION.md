# K-STAR PROJECT アーキテクチャ決定記録

**作成日**: 2026-03-26
**ステータス**: 承認済み

---

## 概要

K-STAR PROJECT（K-POP音楽チャートWebサイト）のデータ管理方式として、**静的JSON方式**を採用する。

---

## 決定事項

### 採用: 静的JSON方式

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  管理ツール   │────→│  JSONファイル │────→│  Vercel/     │
│  (ローカル)   │生成 │  (Git管理)   │Push │  Netlify     │
└──────────────┘     └──────────────┘     └──────────────┘
                                                 ↓
                                          ┌──────────────┐
                                          │  公開サイト   │
                                          │  (静的)      │
                                          └──────────────┘
```

### 不採用: REST API方式

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  管理画面     │────→│  Express     │────→│  SQLite/     │
│  (React)     │ API │  Server      │     │  PostgreSQL  │
└──────────────┘     └──────────────┘     └──────────────┘
```

---

## 選定理由

### 静的JSON方式を選んだ理由

| 理由 | 説明 |
|------|------|
| **シンプルさ** | サーバー不要、フロントエンドのみで完結 |
| **コスト** | Vercel/Netlify/Cloudflare Pages で完全無料ホスティング |
| **高速性** | CDN配信、エッジキャッシュによる高速表示 |
| **セキュリティ** | 攻撃対象となるAPIが存在しない |
| **開発速度** | API設計不要、即座に実装開始可能 |
| **バージョン管理** | JSONをGit管理、全変更履歴が残る |
| **更新頻度** | 週1回程度のチャート更新には十分 |
| **運用規模** | 個人または少人数での管理に適合 |

### REST API方式を選ばなかった理由

| 理由 | 説明 |
|------|------|
| **開発コスト** | サーバー、DB、API設計が必要で初期コストが高い |
| **運用コスト** | 常時稼働サーバーが必要（月$5〜20） |
| **複雑性** | 認証、エラーハンドリング、セキュリティ対策が必要 |
| **オーバースペック** | 現時点の要件に対して過剰な構成 |

## 現在の要件

| 項目 | 決定事項 |
|------|----------|
| **データソース** | 管理ツールからJSON生成 |
| **チャート種類** | 複数（楽曲、アルバム、アーティスト、ストリーミング等） |
| **ユーザー機能** | 不要（閲覧のみ） |
| **更新頻度** | 週1回程度 |
| **管理者** | 個人（単一管理者） |

---

## 静的JSON方式の制約事項

以下の機能は静的JSON方式では**実装不可能**または**困難**：

| 機能 | 制約 | 回避策 |
|------|------|--------|
| リアルタイム更新 | 不可 | Git push後、数分でデプロイ完了 |
| ユーザー認証 | 不可 | 将来必要時にAPI方式へ移行 |
| お気に入り同期 | 不可 | ローカルストレージで代替（デバイス間同期なし） |
| コメント機能 | 不可 | 外部サービス（Disqus等）で代替可能 |
| 検索（大量データ） | 遅い | クライアント側検索、データ量を制限 |
| 複数管理者同時編集 | 困難 | Git運用ルールで対応 |
| 外部API連携 | 不可 | 手動でデータ取得・入力 |

## 将来の拡張オプション

### 段階的移行パス

現在の静的JSON方式から、将来的にREST API方式へ移行することは可能。

```
Phase 1 (現在)     Phase 2 (将来)      Phase 3 (将来)
静的JSON方式   →   ハイブリッド方式  →   フルAPI方式

・JSONデータ        ・静的サイト維持     ・完全なAPI
・Git管理           ・読み取りAPIを追加   ・DB移行
・ローカル管理      ・書き込みは手動      ・管理画面
```

### REST API方式で追加可能な機能

移行した場合に実現可能な機能一覧：

#### 1. 外部データソース連携

| 連携先 | 取得可能データ |
|--------|---------------|
| Spotify API | ストリーミング数、アルバムジャケット、30秒プレビュー |
| Melon/Genie/Bugs | 韓国国内チャート順位 |
| YouTube Music | MV再生数 |
| Apple Music | 楽曲メタデータ |

**実装例**:
```typescript
// 毎日深夜にSpotifyからデータ取得
cron.schedule('0 3 * * *', async () => {
  const tracks = await spotify.getPlaylistTracks('K-POP Hot 50');
  await db.charts.updateStreamingCounts(tracks);
});
```

## 移行判断基準

以下の条件が発生した場合、REST API方式への移行を検討：

| トリガー | 説明 |
|----------|------|
| ユーザー機能の必要性 | ログイン、お気に入り同期が必要になった |
| 外部API連携 | Spotify等との自動連携が必要になった |
| 複数管理者 | 同時編集が必要になった |
| リアルタイム性 | 即時更新が必要になった |
| データ量増大 | クライアント側検索が遅くなった |
| 収益化 | 広告管理、会員機能が必要になった |

## 運用ワークフロー

### チャート更新手順

```
1. ローカルで admin-tool を起動
   $ cd admin-tool && pnpm dev

2. UIでチャートデータを入力・編集

3. JSONファイルを生成（data/ に出力）

4. Git でコミット & プッシュ
   $ git add data/
   $ git commit -m "Update chart: 2025-03-22"
   $ git push

5. Vercel/Netlify が自動デプロイ（2〜3分）

6. 公開サイトに反映
```
